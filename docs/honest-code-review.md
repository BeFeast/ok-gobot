## Hardcoded

- `web/index.html:236-239`. What is wrong: the web UI is hardwired to `ws://127.0.0.1:8787/ws`, and the computed protocol variable is unused. Why this is a problem: custom `control.port`, `wss`, reverse proxying, and tokenized control connections are ignored by the first-party UI. Priority: 🟡 important.
- `internal/tools/tools.go:267-324`, `internal/browser/manager.go:89-92`. What is wrong: privileged tools and paths are hardcoded: `local` is always enabled, Obsidian is auto-discovered in `~/Obsidian`, and Chrome uses `~/.ok-gobot/chrome-profile`. What is wrong: provider-specific identity and browser-like headers are embedded directly in code. Why this is a problem: these integrations depend on brittle upstream behavior and cannot be managed through configuration. Priority: 🟡 important.
- `internal/cli/web.go:51`, `internal/cli/tui.go:20-21`. What is wrong: default web/control addresses are duplicated in CLI code. Why this is a problem: operational parameters are scattered across the codebase instead of coming from one canonical config path. Priority: 🟢 desirable.

## Security

- `internal/control/server.go:71-77`, `internal/control/server.go:153-178`, `internal/control/tui_server.go:115-123`, `internal/config/config.go:198-201`. What is wrong: the control WebSocket is enabled by default, loopback is allowed without token by default, and `Origin` is never checked. Why this is a problem: any malicious webpage opened in the browser can attempt CSWSH against the local control surface and drive the bot. Priority: 🔴 critical.
- `internal/control/protocol.go:103-125`, `internal/control/hub.go:75-94`, `internal/control/tui_bridge.go:59-95`, `internal/bot/hub_handler.go:58-115`. What is wrong: raw tool arguments, tool outputs, approval commands, and run deltas are broadcast to every connected control client. Why this is a problem: one local client can see another chat's prompts, shell commands, tool output, and approval flow. Priority: 🔴 critical.
- `internal/control/server.go:317-331`, `internal/control/protocol.go:116-125`. What is wrong: `approval.respond` is authorized only by global `approval_id`, with no ownership binding to chat, session, or connection. Why this is a problem: any connected client that sees an approval event can approve or deny another client's dangerous command. Priority: 🔴 critical.
- `internal/bot/bot_approval.go:10-14`, `internal/bot/bot_approval.go:23-39`, `internal/app/app.go:318`, `internal/tools/tools.go:99-117`. What is wrong: the approval system is not actually wired into `local`; `InitializeApprovalSystem()` registers callbacks only, while `LocalCommand` runs `bash -c` if `ApprovalFunc` is nil. Why this is a problem: the advertised dangerous-command approval boundary can be absent entirely. Priority: 🔴 critical.
- `internal/tools/tools.go:91-117`, `internal/tools/tools.go:267-268`, `internal/agent/registry.go:40-45`, `internal/agent/registry.go:125-127`. What is wrong: `local` is always registered, executes arbitrary `bash -c`, and the default agent allows all tools. Why this is a problem: this is unrestricted host command execution by default, not a narrowly scoped tool. Priority: 🔴 critical.
- `internal/bot/approval.go:44-99`. What is wrong: dangerous command detection is naive substring matching. Why this is a problem: trivial syntax changes, wrappers, spacing, or shell indirection bypass the approval detector. Priority: 🔴 critical.
- `internal/tools/browser_tool.go:278-316`, `internal/tools/browser_tool.go:368-399`, `internal/browser/manager.go:293-299`, `internal/tools/tools.go:323-324`. What is wrong: the browser tool accepts arbitrary URLs and uses a persistent Chrome profile. Why this is a problem: `file://`, localhost, internal dashboards, and already-authenticated sites become data exfil paths through snapshots, text extraction, and screenshots. Priority: 🔴 critical.
- `internal/tools/tools.go:165-186`, `internal/tools/obsidian.go:127-136`. What is wrong: path validation relies on string prefix checks and does not resolve symlinks. Why this is a problem: a symlink inside the allowed root can escape the intended sandbox and expose files outside workspace/vault. Priority: 🔴 critical.
- `internal/tools/web_fetch.go:26-33`, `internal/tools/web_fetch.go:54-70`, `internal/tools/web_fetch.go:174-211`. What is wrong: SSRF validation is performed on the initial URL only; redirects are not revalidated. Why this is a problem: a public URL can redirect to loopback, metadata endpoints, or private infrastructure and bypass the SSRF guard. Priority: 🔴 critical.
- `web/index.html:7-9`, `web/index.html:431`, `web/index.html:442`, `web/index.html:450`. What is wrong: the web UI loads third-party JS from CDN without SRI/CSP pinning and pipes `marked.parse(...)` directly into `innerHTML`. Why this is a problem: model output or tool output can become DOM XSS, and the frontend also has a supply-chain exposure. Priority: 🔴 critical.

- `internal/app/app.go:127-139`, `internal/bot/config_reload.go:28-48`. What is wrong: hot reload updates `a.config` and reports success, but it does not rebuild the running bot, auth manager, tool registry, AI client, control server, or API server. - `internal/app/app.go:95-98`, `internal/app/app.go:221-225`. What is wrong: `SpawnSubagent` is a stub that sends a chat message, and cron execution is still a `TODO`. Why this is a problem: parts of the public surface claim capabilities that are not actually implemented. Priority: 🟡 important.
- `internal/control/protocol.go:5-27`, `internal/control/hub.go:90-94`. What is wrong: legacy control protocol and TUI protocol are mixed on the same WebSocket without negotiation. - `internal/control/hub.go:42-73`. What is wrong: the hub run loop has no shutdown path. Why this is a problem: component restarts and tests leak goroutines. Priority: 🟢 desirable.
