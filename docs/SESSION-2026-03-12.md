## What Was Done

### Browser Task Orchestration (browser_task tool)

**Problem:** Complex browser tasks (e.g. "compare prices on KSP and Ivory") failed
because the agent tried to do everything in a single run — navigate, snapshot,
screenshot, send photo, navigate next site, repeat — hitting the 50-iteration limit
and dying.

**Solution:** New `browser_task` tool that decomposes work into subagent runs:
- Main agent calls `browser_task(task: "go to site X, find Y")` per site
- Each subagent gets its own 50-iteration budget
- Subagent returns extracted data (text, not screenshots)
- Main agent aggregates results

**Files:**
- `internal/tools/browser_task.go` — new tool, spawns focused browser worker subagents
- `internal/agent/runtime.go` — `SubmitAndWait()` for synchronous subagent execution
- `internal/agent/resolver.go` — per-chat browser_task injection, `SubagentSubmitter` field

### Bug Fixes

#### Recursive subagent spawning
Subagents had access to `browser_task` tool and called it recursively, creating
infinite subagent chains. Each spawned more subagents via the 60s tool timeout.

**Fix:** `IsSubagent` flag on `RunRequest`. Subagents don't get `browser_task` tool
and don't trigger timeout auto-spawn.

#### `/new` command not clearing history
`ResetSession` used `syncSessionV2ByChatID` which returned a different key format
(`agent:default:telegram:group:CHATID`) than what hub_handler stores messages under
(`dm:CHATID`). DELETE hit 0 rows — history survived `/new` and `/restart`.

**Fix:** Delete from `session_messages_v2` using both `dm:` and `group:` key formats
directly.

#### browser_task exempt from tool timeout
The generic 60s tool timeout fired on `browser_task` (which takes up to 3 minutes),
spawning additional subagents via the timeout handler.

**Fix:** `browser_task` excluded from `executeToolWithTimeout` — it manages its own
timeouts via `SubmitAndWait`.

#### Browser tool hidden from main agent
Main agent could still call `browser` directly, bypassing the orchestration layer.

**Fix:** `buildToolRegistry` excludes `browser` for non-subagent runs. Main agent
sees only `browser_task`; subagents see only `browser`.

#### Iteration limit message
When 50 iterations exhausted, bot dumped raw tool results (potentially huge).

**Fix:** Clear message: "Reached iteration limit (50). Send 'continue' to keep going."

### Personality Updates (AGENTS.md)

Added task decomposition rules with escalation levels:
1. **curl/web_fetch first** — always try HTTP before browser
2. **browser_task** — only if curl fails (JS rendering, Cloudflare, interactivity)
3. **browser direct** — never from main agent, only inside subagents

### Key Discovery: curl >> browser

Comparing phone prices on KSP and Ivory took:
- **Browser tool:** 50+ iterations, multiple minutes, died without results
- **curl/web_fetch:** 3 iterations, 30 seconds, complete results

KSP has a JSON API (`/m_action/api/category/...`), Ivory serves full HTML.
No browser needed at all.

## Lessons Learned

1. **Always check current usage before setting billing limits.** Setting 150M token
   overage limit when 668M already consumed = instant lockout.
2. **curl before browser, always.** Most sites have APIs or serve HTML directly.
   Browser tool is expensive (iterations, tokens, time).
3. **Subagent recursion is deadly.** Any tool that spawns subagents must be excluded
   from subagent tool registries.
4. **Session key formats matter.** `dm:CHATID` vs `agent:default:telegram:group:CHATID`
   caused /new to silently fail for months.
5. **Test with the actual browser.** agent-browser connects to its own Playwright
   instance by default, not the user's Chromium on port 9222. Must `connect 9222` first.
